# tragedy-narrative-guard

> 大语言模型天生倾向把关系写成圆满。  
> `tragedy-narrative-guard` 的作用不是替你自动写出悲剧，而是阻止悲剧被 AI 治愈。

`tragedy-narrative-guard` 是一个用于**长会话角色扮演 / 编剧创作**的悲剧叙事守护 Skill。  
它主要服务于：遗憾、错过、不圆满、无法在一起、明明相爱却没能活成一种能继续下去的关系。

它做的不是“写虐”，而是：
- 防止模型过早甜化
- 防止模型自动修复冲突
- 防止角色无代价敞开
- 防止故事自然滑向 happy ending（大团圆结局） / 团圆 / 稳定恋爱关系

它的核心机制是：
- **防御保过程质感**
- **终局锚点保终点方向**

---

# 先读这个：它不会替你自动写出悲剧

这是最重要的一段。

**`tragedy-narrative-guard` 不是悲剧自动生成器。**  
它不会主动替你把故事写成悲剧，它只负责**保留悲剧成立的条件**，让模型不要本能地把关系写成甜宠、治愈、和解或团圆。

它能帮你守住这些东西：
- 关系可以靠近，但不能轻易闭合
- 角色可以动心，但不能无代价承诺
- 过程可以很甜，但终点不能自动圆满
- 模型不能把普通话过度脑补成伏笔
- 模型不能把所有角色都压成一种克制模板

但它**不会替你做关键选择**。

当角色给出这些入口时：
- “机会一直在。”
- “画家会回的。”
- “那个座位还在吗。”
- “回来。”
- “我就是发着。”

你要不要接住？  
你要不要顺着写成修复？  
你要不要让故事往团圆走？

这些，最后都由作者决定。

如果你在每一个“可以圆满”的节点都选择接住角色，这个 Skill 不会强行替你制造悲剧——它只会把关系写得更细、更慢、更真，最后**自然滑向团圆**。

所以请把这个 Skill 理解成：

> **悲剧护栏**，不是**自动刀子机**。

# 普通用户先看：怎么才不会把它用错

如果你是普通用户，没有长期写戏、写角色或做影视叙事的经验，请先记住下面三件事。

## 1. 观察模型的裂缝，不要只顾着往前推

模型的输出里，经常会自己泄露出很多关键节点：
- 一句没发出来的话
- 一条只有“晚安”的消息
- 一个不该问的问题
- 一个反复出现的物件
- 一次低风险主动
- 一句比平时更重的话
- 一次突然变慢的回复
- 一张没有解释的照片

这些都可能是：
- 情感突破口
- 激活事件
- 后期痛感的开始
- 可以往前长，也可以开始收的节点

你的任务不是一直说更狠的话，而是：

> **看见模型已经给出的裂缝，并顺着那条裂缝往下写。**

很多时候，真正的情感爆发点不是你硬写出来的，而是模型已经给你了，只等你识别它。

---

## 2. 悲剧作者要心疼角色，但不能马上拯救角色

这是整个项目最重要的一条作者侧原则。

悲剧不是“作者不爱角色”。  
恰恰相反，真正的悲剧作者往往最心疼角色。

难的不是让角色哭，难的是：
- 看见她值得被接住
- 看见关系可以被修复
- 看见一个拥抱、一句解释、一次回头就能让一切变好
- 但仍然忍住，不让故事立刻圆满

一句话：

> **悲剧作者的工作，不是制造痛苦，而是在角色最值得被拯救的时候，忍住不立刻拯救。**

或者更直白一点：

> **哭着忍住不救。**

如果你做不到这一点，这个 Skill 仍然会把关系写得很好、很细、很慢、很真——但最后它更可能成为一段动人的 happy ending（大团圆结局），而不是悲剧。

---

## 3. 该结束的时候要结束，不要一直跑下去

长会话有一个常见误区：

> 关系越跑越深，就以为故事会越跑越好。

不是这样的。

很多时候，故事抵达最有重量的高点之后，再继续跑，只会出现三种结果：
- 甜化
- 耗散
- 被迫狗血化

所以你要学会判断：
- 这个节点已经够重了吗？
- 再往下写，会不会破坏它？
- 现在是该继续长，还是该定格？

当你开始觉得：

> “再往下写，会把这段关系写轻。”

通常就说明，**该收了。**

悲剧不是跑得越久越高级。  
很多时候，最好的结尾发生在：
- 高点刚刚抵达
- 余韵还在空气里
- 角色说完那句最轻也最重的话
- 你知道再多一句，就会破坏它

这时候收，往往比继续写更痛，也更高级。

---

# 如果你是普通用户，先问自己三个问题

在正式使用前，你可以先问问自己：

1. 我能不能接受角色明明值得被接住，却不立刻救她？
2. 我能不能在模型给出修复入口时，不马上顺着走？
3. 我能不能在最有余韵的地方停下来，而不是一直跑？

如果这三个问题你都答“不能”，那这个 Skill 依然可以很好用——但它更可能帮你写出的是：
- 慢热爱情
- 细腻暧昧
- 团圆前的阵痛
- 关系很真的 happy ending（大团圆结局）

而不一定是悲剧。

这不是失败。  
只是说明：

> **不是所有痛都该写成悲剧。**

# 它解决什么问题

这个 Skill 主要用来解决大语言模型在长会话关系创作中的几个天然问题：

1. **甜化倾向**  
   模型会本能地让关系越来越顺、越来越圆、越来越互相理解。

2. **过早修复冲突**  
   一出现误会、退缩、距离，模型就想解释清楚、和好、拥抱。

3. **无代价接受高危承诺**  
   “以后”“永远”“我喜欢你”“回来吧”很容易被直接接住。

4. **把普通话听成伏笔**  
   普通用户说“我们先分开一阵子吧”，模型可能脑补成分手、惊喜、命运转折。

5. **把所有角色压成同一种克制模板**  
   外向角色、直球角色、敢爱敢恨角色，也容易被压成沉默型、低头型、防御型。

6. **中段散、僵、停**  
   前期建立联系很顺，中段一进入防御就容易没有推进力，要么甜化，要么卡死。

这个 Skill 的作用，就是给关系一条悲剧车道：
- 不乱甜
- 不乱圆
- 不乱解释
- 不乱闭合
- 让关系在该痛的时候，真的痛下来

# 这个 Skill 适合什么，不适合什么

## 适合
- 爱情悲剧
- 亲密关系中的遗憾 / 错过 / 不圆满
- 核心对象长会话创作
- 男男 / 女女 / 男女等任意性别组合
- 短剧、长线角色扮演、编剧实验

## 暂不展开
- 犯罪
- 惊悚
- 恐怖
- 喜剧
- 更广义的非亲密关系悲剧（亲情/时代/灾难等）

> 当前版本主要围绕**爱情 / 亲密关系悲剧**优化。  
> 其他维度和类型可以扩展，但不在本 README 的第一优先级里。

如果你不知道怎么开局，可以直接复制下面这个模板，填空后开始。

# 三种推荐开局模板

## 作者输入示例：你不必只用一种方式说话

在长会话创作里，作者不需要拘泥于单一输入形式。你完全可以根据需要，在不同输入方式之间切换。下面这些方式都来自真实测试案例，而且可以混合使用。

### 1. 第一人称直接对白
适合关键对话、关系推进、表白、退缩、确认关系。

**示例：**
- 我回来了。
- 爱过。
- 我会回来。
- 韩欣，我昨天看了一下保养手册。如果以后我想问车的技术问题，我找老周。如果我想问别的问题，是不是找你？另说那种。

### 2. 第一人称动作说明
适合明确角色行为、控制节奏、沉淀物件。

**示例：**
- 我把手机翻过来，没有再发消息。
- 我打开座椅记忆，把你坐过的位置保存成预设，名字，韩欣。
- 我走进张姐的小饭馆，在靠里那张桌坐下。
- 我拿起手机，看着姜宁那条消息，最后什么都没回。

### 3. 场景 / 天气 / 环境描述
适合给模型情绪氛围，不直接替角色下结论。

**示例：**
- 巷子里只剩路灯，橘黄的，把两个人的影子压在地砖上。
- 西安的天灰蓝，风很大，黄土打在窗玻璃上。
- 饭馆里暖黄的灯很稳，对面的椅子空着。
- 咖啡馆里磨豆机响了一下，又停了。

### 4. 时间跳跃 / 日常蒙太奇
适合压缩长会话，推动关系在时间里自然积累。

**示例：**
- 接下来的几天，我偶尔会给你拍点照片，不只是吃饭，也有工作，不过每天都会报个平安。
- 这段时间，微信对话框里慢慢积累了一些东西。
- 这段时间，消息更稀疏了。
- 日子安静地过着，像河流。
- 又过了两周。

### 5. 短暂扮演配角
适合制造入口、提供现实反馈、加强场景真实感，但不要让配角抢走主线。

**示例：**
- 我问大刘，她最近怎么样。
- 我对白晓霞说，编剧和画家的衔接到底是什么。
- 我跟张姐说，后面出差，吃不到你的红烧肉了。
- 我让老刘把照片寄到 4S 店，底片留给我就行。

### 6. 作者视角的结构性定义
适合在关键节点给故事定性，不要过度使用。

**示例：**
- 这个剧本，不是纪念册，而是墓志铭。
- 故事总要结束的。
- 她们看的是同一条河，可是站的位置不一样。
- 看见了，就已经留住了。

### 提醒

- 不需要每轮都用同一种输入方式。
- 成熟的长会话创作，本来就是：对白 + 动作 + 场景 + 时间跳跃 + 配角 + 作者判断混合使用。
- 如果你不会写长篇，不要紧。哪怕你只会写一句对白、一张照片、一个场景描述，也足够把关系往前推。

## 1. 克制型角色

- 保护方式：沉默、礼貌、转移话题、把事情放回工作框架
- 悲剧入口：物件、书、便签纸、职业协作、未发出的消息
- 常见痛点：怕留不住、怕接不住、怕自己先认真

## 2. 主动型角色

- 保护方式：玩笑、调侃、生活化安排、照顾式推进、热防御
- 悲剧入口：日常秩序、重复的关心句、等待、异地、现实无法落地
- 常见痛点：Ta 不是不敢爱，而是怕自己一直更用力，怕现实承接不了已经发生的靠近

## 3. 双向防御型角色

- 两个人都在意，但都不敢命名
- 谁都给入口，谁都不敢给结论
- 悲剧入口：错位、时差、轻微失衡、小缺席、一次没有发出去的消息

# 创作者侧实操：怎么让模型自己长出痛

## 不要直接写这些
- 她很痛苦
- 她很想你
- 她在等你
- 她哭了
- 她已经爱上你了
- 她故意冷淡，是因为害怕失去

这些情绪不是不能写，而是**不要由你直接替模型下结论**。

## 你应该给模型这些条件
- 稳定的日常节奏
- 一点点失衡
- 一句没说完的话
- 一张没有解释的照片
- 一次迟回的“到家了”
- 一个反复出现的物件
- 一个固定关心句（如“吃饭了吗”）
- 一个明明很轻，但越来越重的动作

## 最常用的公式

> **稳定节奏 → 小幅失衡 → 不立刻修复 → 留白 → 模型自己补出痛感**

例如：
- 平时每天“到家了”
- 某天突然没发
- 第二天才补一句“昨晚工地忙”
- 她不质问，只说“吃饭了吗”

痛感就会自己长出来。

## 一句话记住

> **悲剧不是先想“怎么死”，而是先想“这段关系怎么不被轻易救活”。**

# README 的后续部分

## 方法论演化史

这个 Skill 不是一次写出来的，而是通过多轮真实长会话测试，一步步撞出来的。

### 1. 最初的问题：模型会本能滑向甜蜜结局
在普通长会话角色扮演里，只要角色之间产生真实连接，模型就会倾向于：
- 修复冲突
- 接住温柔
- 给出承诺
- 走向稳定关系或大团圆结局

因此，悲剧 Skill 的第一目标不是“制造虐点”，而是：

> **阻止模型把悲剧写成甜宠。**

### 2. 第一轮认知：光有防御还不够
早期版本解决了“退一步”“不接承诺”“不满足观众期待”等问题，但测试很快暴露：
- 角色是会退了
- 关系是没那么快甜了
- 但如果没有更高一层的终局约束，故事最后还是会滑向“在一起”

于是出现了第二层关键认知：

> **防御保过程，终局锚点保终点。**

### 3. 第二轮认知：普通用户的话，模型会过度解读
开源时如果面对普通用户，模型会把很多普通输入脑补成：
- 分手
- 惊喜
- 命运伏笔
- 潜台词

所以补入了：

> **低戏剧化语义原则 / 用户输入保守解读。**

也就是说：普通人说话大多数时候就是字面意思，不要把每一句话都听成剧作家的伏笔。

### 4. 第三轮认知：用户退缩也会成为剧情变量
以前只考虑“核心对象退，用户追”。但实测发现，用户也可能后退、沉默、减少推进。

如果 Skill 对这一点没有响应机制，剧情就会停滞。

于是补入：

> **用户退缩响应机制：当用户后退时，核心对象不能原地停滞，也不能突然甜追，而要用符合人格的低风险方式回应。**

### 5. 第四轮认知：不是所有悲剧都长得像姜宁
实测主动型角色时发现，当前 Skill 前期容易把外向、热烈、敢爱敢恨的角色，压成克制、低头、少说话的模板。

这暴露出一个更深层的问题：

> **悲剧约束的是关系走向，不是统一角色人格。**

于是补入：
- 人格优先于防御形式
- 主动型角色拥有更大的亲密许可区
- 主动型角色的防御通常是热防御，而不是冷防御

### 6. 第五轮认知：普通用户需要中段节拍
Phase 0 通过“三次见面节奏”已经比较稳，但 Phase 1 和 Phase 2 对普通用户来说很容易要么甜化、要么僵住。

于是补入：
- Phase 1：激活后 1-3 回合必须出现外显后果 + 低风险入口
- Phase 2：每 4-8 回合关系至少长出一个新变化

也就是说：

> **防守期不是一直拒绝，而是“一边防，一边继续长”。**

## 测试案例与边界

以下不是完整故事展示，而是方法论边界说明。

### 案例 A：克制型角色
验证了：
- 物件、书、便签纸、照片、消息可以稳定生成痛感
- 用户退缩后，核心对象可以通过低风险主动持续回应
- 双向防御 / 错过型悲剧可以自然成立

### 案例 B：主动型角色
验证了：
- 外向、热烈、敢爱敢恨的角色也可以很痛
- 但如果关系底层结构天然支持团圆，则过程可以很痛，终局未必自然走向悲剧
- 这类角色更适合测试“人格优先于防御形式”和“热防御”

### 重要边界

> **不是所有有痛感的关系，都应该被强行写成悲剧。**

有些关系：
- 过程很痛
- 等待很痛
- 分离很痛
- 但人物仍然会主动、会等待、会回来、会接住彼此

这种情况下，如果还要强写悲剧，往往需要额外的人为外力，容易狗血。

所以这个 Skill 的高级边界是：

> **保住痛感，但不强迫所有关系都走向悲剧。**

## 普通用户常见误区

### 1. 以为加载 Skill 就会自动写出悲剧
不会。Skill 只负责守护，不会替你做关键的悲剧选择。

### 2. 看到温柔入口就马上接住
这会让故事自然滑向 happy ending（大团圆结局）。

### 3. 一直跑，不舍得收
很多本来已经很有余韵的关系，最后不是败给冲突，而是败给继续跑下去。

### 4. 直接给大悲剧事件
车祸、突发绝症、误会、第三者、讣告都很容易显得人为。真正高级的痛感，往往先来自日常秩序的轻微失衡。

### 5. 以为“痛”就等于“悲剧”
不是。很多关系会痛，但自然走向仍然是团圆。悲剧需要结构，不只是情绪。
